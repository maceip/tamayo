Start with the crypto, not the demo.

The Pages demo shows tokens because they are easy to see. The repo also ships cgo-free Go primitives you can import directly: signatures, blind-signature machinery, and the finite-field code underneath them.

PrimitiveWhat it isStandalone useStatus
FAEST A post-quantum signature family built from symmetric primitives and zero-knowledge proofs. Import the Go package to sign runtime statements, policy files, or transparency-log heads. NIST additional-signature round 3 candidate.
MAYO A compact multivariate signature family over GF(16). Use MAYO signing directly, or use its preimage path inside blind signatures. NIST additional-signature round 3 candidate.
PoMFRIT A One-More-MAYO blind-signature construction with verifier checks over the same Go primitives. Build anonymous one-use credentials or other blind-issuance protocols. Implemented and cross-checked against reference vectors in this repo.
GF(16) and binary fields Small and large finite-field arithmetic used by MAYO, FAEST, VOLE checks, and circuit code. Reuse the field math for tests, experiments, or other proof systems without importing the token service. Plain Go implementations with focused algebra and reference-vector tests.
ML-DSA The standardized module-lattice signature used where a FIPS-track signature is the right tool. Bind holder presentations or sign email-token profiles without tying them to blind issuance. Implemented with NIST ACVP-style vectors in the repository.

What the TEE does

A TEE does not make a token private and it does not replace the cryptography. It gives the issuer a measured runtime claim: this request came from code with a known measurement, under an allowed deployment profile.

What policy does

The policy language decides whether issuance is allowed before minting. It can require token family, origin, email evidence, runtime measurement, assurance level, and per-bucket budgets.

The technical part answers the question the story creates.

Why not just give the agent a password? Why does this not become another tracking ID? Why would a hard-mode service trust the pass at all?

Blind issuance

The issuer can approve a sealed pass without learning the final pass that a verifier will later see.

Zero knowledge

The holder can prove the pass is valid without dumping the extra facts that were checked along the way.

Policy before minting

A service can require email evidence, runtime measurement, or another rule before a pass is issued.
